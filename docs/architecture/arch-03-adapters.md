# arch-03: The Adapter Model

Kirimana's core is platform-agnostic. It does not know what Databricks is, what Iceberg is, or how to write a Spark SQL `CREATE TABLE`. Everything platform-specific lives behind typed Python protocols called adapters. This document describes the two-category adapter model, the adapters currently implemented, how surface versioning works, and what is required to add a new one.

## Two categories

Kirimana distinguishes two top-level adapter categories, each with its own protocol surface:

- **PlatformAdapter** — owns the physical data substrate. It knows how to apply a contract against a real warehouse: create tables, evolve columns, write data, read back state for verify. PlatformAdapters bind to a specific lakehouse or warehouse (Databricks, Iceberg, Fabric, MSSQL).
- **FrameworkAdapter** — owns a transformation framework that sits on top of a platform. It compiles silver/gold contracts into framework-native artefacts (dbt models, SQL projects). FrameworkAdapters are platform-aware but not platform-bound; a dbt FrameworkAdapter can run on top of any compatible PlatformAdapter.

Two further protocol categories rounded the model out as the system matured:

- **OrchestrationAdapter** — compiles schedules and DAGs into platform-native jobs (Databricks Workflows, Kubernetes CronJobs). Kirimana compiles; the platform's scheduler executes.
- **CatalogBackend** — pluggable catalog persistence (SQLite default, Postgres, external catalog pull — see arch-05).

All four are discovered through the same plugin-registry mechanism. A user installs an adapter as a Python package; Kirimana finds it by entry point.

## Implemented adapters

In v0.9 preview, the following PlatformAdapters are present in the tree at `packages/adapters/dca_adapters/backends/`:

- **localduckdb** — DuckDB for local development and CI. Production-ready for evaluation; not intended as a production target.
- **databricks** — Databricks SQL Warehouses + Unity Catalog. The flagship target; production-ready for the use-cases documented on the maturity page.
- **iceberg** — direct Iceberg via `pyiceberg`, peer to Trino. Beta.
- **trino** — Trino + Iceberg + Polaris stack for cost-effective open lakehouse deployments. Beta.
- **fabric_dw** / **fabric_lakehouse** — Microsoft Fabric, two flavours. Beta.
- **mssql** — SQL Server, primarily for source-system migration workflows. Beta.
- **postgres** — Postgres as a small-scale target (Starter edition default). Beta.

FrameworkAdapters: **dbt-core** is the first and currently the only one. It is a bridge — Kirimana does not fork dbt; it generates dbt project shape that dbt-core executes.

## Surface versioning and conformance

Every adapter protocol carries a `surface_type` (`plugin` for third-party-implementable, `internal-protocol` for core-only) and a semantic version. Adapters declare a capability manifest naming which surface versions they implement.

A conformance test suite ships with the core (`packages/adapters/tests/`). Each adapter category has a fixture pack: a series of canonical contracts, expected plan outputs, and expected post-apply states. To be eligible for the registry, an adapter must pass the suite at the declared surface version. The conformance result is part of the plan signature in arch-02 — a plan generated by adapter version 1.3.0 is not interchangeable with one generated by 1.4.0 even on identical contracts.

Deprecation follows a two-MINOR window. When a surface adds a method, adapters declaring the older version continue to load; when a surface removes one, two MINOR releases must pass before the removal lands.

## How to add an adapter

The shape:

```python
from dca_adapters.protocols import PlatformAdapter

class MyPlatformAdapter(PlatformAdapter):
    surface_version = "1.4.0"
    capabilities = {"merge_into": True, "column_masks": False, ...}

    def plan(self, contracts, current_state) -> Plan: ...
    def apply(self, plan) -> ApplyResult: ...
    def verify(self, contracts) -> VerifyResult: ...
    def read_state(self) -> PlatformState: ...
```

Register as an entry point under `dca.platform_adapters` (FrameworkAdapters register under `dca.framework_adapters`). Run the conformance suite. Submit a manifest (`*.manifest.yml`) declaring capabilities. The adapter is then discoverable to any Kirimana installation that installs the package.

The same pattern, mutatis mutandis, applies to FrameworkAdapter, OrchestrationAdapter, and CatalogBackend.
