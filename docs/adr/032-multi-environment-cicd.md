# ADR 032 — Multi-environment CI/CD with git-SHA as release ground truth

- **Status:** Accepted
- **Date:** 2026-05-13
- **Re-curated from:** original ADR 0017 (archived)

## Context

A Kirimana project's `dca.yml` declares `targets: { dev, test, prod }` with per-environment catalogs, vault configs, and warehouse paths. The project commits to one catalog per target (ADR 031), so three environments mean three catalogs.

What the platform did not yet have:

- A way to ask *"which commit is live in test right now?"*
- A way to diff *"HEAD vs prod"* before promoting
- A way to detect environment drift (someone applied an out-of-band change)
- A versioning convention for generated artefacts (dbt files, silver SQL, future orchestration bundles) — they're deterministic per the apply pipeline, but no convention stamped which generator produced them

The user requirement was explicit: dev/test/prod at MVP, version every artefact Kirimana creates, keep environments in sync with a reliable promotion story.

Three candidates were considered:

1. **Git-SHA as release ground truth + content-addressed generated artefacts.** Every apply stamps the commit SHA into a manifest row; generated files are deterministic derivatives of `(contract content × generator version)` so they inherit identity from the SHA.
2. **Separate release registry service** — a Kirimana-owned API tracking releases outside git. Rejected: violates simplicity and re-invents git.
3. **Per-environment branches** — `env/dev`, `env/test`, `env/prod` as long-lived branches. Rejected: merge conflicts become environment conflicts and SHA identity becomes fuzzy.

Option 1 was accepted.

## Decision

### Git is the ground truth for everything Kirimana applies

Every `kiri apply --target <env>` stamps a row into the existing `_dca_meta._dca_project_manifest` table in the target's catalog. Release tracking reuses this table rather than introducing a new one — the columns relevant to release ground truth are:

- `release_sha` — git commit SHA (40 hex)
- `release_tag` — semver tag if present (`v0.5.2-rc1`), else `null`
- `target_env` — `dev` | `test` | `prod`
- `generator_versions` — JSON object of per-adapter generator versions
- `applied_at`, `applied_by`, `apply_status`

### Generated artefact versioning

Every generator (dbt-bridge, silver SQL emitter, orchestration compiler) declares its version. Generated files include a header:

```
# Generated by kirimana-dbt-bridge v1.4.2
# From contract sha256:abc123... at git-sha 4f2e...
# DO NOT EDIT — regenerate with `kiri apply`
```

### Six promotion commitments

1. **Forward-only.** `dev → test → prod`. No skipping, no jumping back. (Linked to ADR 010.)
2. **Same SHA across environments at GA time.** A release is "live in prod" only if `prod.release_sha == test.release_sha == dev.release_sha` for at least 24h before stamping a release tag.
3. **Drift detection on every apply.** Plan-time computes the diff between local SHA and current target SHA; any divergence requires explicit `--allow-drift`.
4. **Materialised data versioning.** Silver/gold materialisations are tagged with the SHA that produced them (via Delta table properties or platform equivalent); reverting a deploy reverts data state if the materialisation strategy supports it.
5. **CI gate per environment.** Promotion to `test` requires CI green on the source commit; promotion to `prod` requires test-environment manifest to match the source commit for ≥1h.
6. **No backdoor applies in prod.** `kiri apply --target prod` rejects any commit not tagged or not on `main`.

## Consequences

- **Positive:** Operators always know what's live. Promotion is auditable. Rollback is a `git revert` + re-apply, not a database surgery operation.
- **Negative:** Discipline cost — the team must use the manifest, not bypass it. CI gates add latency to hotfix flows; an `emergency` flag exists but logs to the audit trail.
- **Neutral:** Backports and cherry-picks remain valid but require explicit tagging.
