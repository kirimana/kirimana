## Adding maintainers

Kirimana transitions from Phase 1 (BDFL) to Phase 2 (maintainer team) when **≥ 3 sustained contributors** each accumulate **≥ 6 months of merged work** and are nominated by an existing maintainer (initially the BDFL). At that point a new ADR is written, this file is updated, and the lazy-consensus model takes effect.

Until that trigger is met, this file lists one maintainer.

## Removing maintainers

Maintainers may step down by opening a PR against this file. Involuntary removal (e.g., sustained inactivity, Code-of-Conduct violation) follows the process defined in `GOVERNANCE.md` for the then-current phase.

## Contacting maintainers

- **General technical questions:** open a GitHub issue
- **Security disclosures:** follow [`SECURITY.md`](SECURITY.md) — do not email maintainers directly for vulnerabilities
- **Governance or conduct matters:** email the BDFL (Phase 1) or the maintainer team (Phase 2+)
